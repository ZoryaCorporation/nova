# Nova Error System Specification

Nova's error system is designed to do more than report failures — it is
designed to *teach*. When something goes wrong, whether at compile time
or runtime, Nova produces rich, color-coded diagnostics with source
context, caret underlines, and — most distinctively — NLP-inspired
"did you mean?" suggestions and contextual help notes derived from
edit-distance analysis and source-line pattern matching.

The system is built from three layers. The **diagnostic renderer**
formats errors with ANSI colors, source windows, and caret lines. The
**error catalog** maps 57 classified error codes to short names and
multi-paragraph explanations accessible via `nova --explain E1001`. The
**suggestion engine** applies Levenshtein edit distance, KWIC (Key Word
In Context) source analysis, and pattern-based hint generation to attach
actionable sub-diagnostics to every error.

This document describes the architecture and implementation of Nova's
error system as of v0.2.0. It covers the error code taxonomy, the
diagnostic data structures, the rendering pipeline, the NLP-inspired
suggestion engine, and the integration points where errors are
classified and enriched. For the virtual machine that raises these
errors, see the [VM Specification](NOVA_VM_SPEC.md). For the bytecode
instruction set, see the [Bytecode Specification](NOVA_BYTECODE_SPEC.md).

## 1. Design Philosophy

Most scripting languages treat errors as afterthoughts: a message
string, a line number, and maybe a stack trace. Nova treats errors as
a first-class feature of the developer experience.

The guiding principles are:

**Errors should be precise.** Every error carries a classified code
(E1001, E2013, E3005) that identifies exactly what went wrong. The code
taxonomy separates compile-time errors (E1xxx) from runtime errors
(E2xxx) and I/O or system errors (E3xxx). Fine-grained codes like E2013
("arithmetic type error") distinguish a string-plus-number mistake from
a nil dereference, even though both are "type errors" in the broad
sense.

**Errors should show context.** Every diagnostic includes the source
line where the error occurred, a caret underline marking the offending
token, and a context window of surrounding lines. The developer sees
exactly what the compiler or VM was looking at when things went wrong.

**Errors should teach.** Beyond the error message, Nova attaches
*sub-diagnostics* — help notes and suggestions — that explain what to
do. A "did you mean?" hint when you misspell `prnt` as `print`. A
"use tonumber()" suggestion when you try to add a string and a number.
A note that `==` works across types when you get a comparison error.
These hints are generated by an NLP-inspired engine that applies
Levenshtein edit distance and source-line pattern matching.

**Errors should be beautiful.** Nova uses a carefully designed ANSI
color palette — hunter green accents, red error labels, sky blue line
numbers, emerald highlights — that makes diagnostics easy to scan in a
terminal. The palette respects the `NO_COLOR` environment variable for
accessibility.

**Errors should be explorable.** Every error code has a long-form
explanation accessible via `nova --explain E1001`. These explanations
are multi-paragraph tutorials with examples and suggested fixes,
designed to teach beginners and refresh experts.

## 2. Error Code Taxonomy

Nova classifies errors into four tiers, each with a dedicated numeric
range. Every error code is a constant in the `NovaErrorCode` enum.

### 2.1 Compile-Time Errors (E1001–E1021)

These errors are raised during lexing, preprocessing, parsing, or
compilation — before the VM executes a single instruction. They
represent syntactic or structural problems in the source code.

| Code | Name | Description |
|------|------|-------------|
| E1001 | unexpected token | The parser encountered a token that doesn't fit the grammar |
| E1002 | undefined variable | Reference to a name not found in any scope |
| E1003 | too many decs | Exceeded the dec variable limit per function |
| E1004 | too many registers | Exceeded the register file capacity (256 per function) |
| E1005 | too many scopes | Exceeded the nesting depth for scopes |
| E1006 | too many functions | Exceeded the nesting depth for function definitions |
| E1007 | invalid assignment | Left-hand side of `=` is not a valid target |
| E1008 | break outside loop | `break` or `continue` statement outside a loop body |
| E1009 | duplicate label | A `::label::` with the same name already exists in this scope |
| E1010 | unresolved goto | A `goto` target label doesn't exist in this scope |
| E1011 | too many arguments | Exceeded the argument limit for a function call or definition |
| E1012 | too many fields | Exceeded the field limit in a table constructor |
| E1013 | constant overflow | Exceeded the constant pool limit for a function |
| E1014 | expected token | Expected a specific token (e.g., `)` after parameters) |
| E1015 | interpolation overflow | String interpolation expression too complex |
| E1016 | expected function | Expected a function definition after `async` keyword |
| E1017 | out of memory | Compiler ran out of memory during compilation |
| E1018 | too many labels | Exceeded the label limit per function |
| E1019 | too many gotos | Exceeded the pending goto limit per function |
| E1020 | module name overflow | Module name string is too long |
| E1021 | too many parameters | Exceeded the parameter limit for a function definition |

### 2.2 Runtime Errors (E2001–E2025)

These errors are raised during VM execution — type mismatches, nil
accesses, failed operations, and other dynamic faults. This is the
richest category, with fine-grained codes that distinguish arithmetic
type errors from comparison type errors from concatenation type errors.

| Code | Name | Description |
|------|------|-------------|
| E2001 | type error | Generic type mismatch (fallback) |
| E2002 | stack overflow | Call stack exceeded the maximum depth |
| E2003 | out of memory | Runtime memory allocation failure |
| E2004 | nil dereference | Attempted to use nil where a value is required |
| E2005 | division by zero | Division or modulo with a zero divisor |
| E2006 | invalid operand | Operand type not valid for the operation |
| E2007 | index out of bounds | Array or string index outside valid range |
| E2008 | call of non-function | Attempted to call a nil, number, string, or other non-function value |
| E2009 | wrong argument count | Function called with incorrect number of arguments |
| E2010 | coroutine error | Invalid coroutine operation (resume dead, yield from main) |
| E2011 | metamethod error | Metamethod returned invalid result or caused error |
| E2012 | general runtime error | Catch-all for errors that don't fit other categories |
| E2013 | arithmetic type error | Arithmetic operator applied to incompatible types |
| E2014 | comparison type error | Ordering operator applied to incompatible types |
| E2015 | concatenation type error | `..` operator applied to non-string/non-number value |
| E2016 | length type error | `#` operator applied to non-string/non-table value |
| E2017 | negation type error | Unary `-` applied to non-number value |
| E2018 | index on non-table | `.` or `[]` access on a nil, number, or other non-table value |
| E2019 | invalid key type | Table key is nil or NaN |
| E2020 | for loop type error | Numeric `for` loop bounds are not numbers |
| E2021 | argument type error | Standard library function received wrong argument type |
| E2022 | argument count error | Standard library function missing a required argument |
| E2023 | module not found | `require()` could not locate the specified module |
| E2024 | metamethod chain too deep | Metamethod lookup exceeded the depth limit |
| E2025 | bitwise on non-integer | Bitwise operator applied to non-integer value |

### 2.3 I/O and System Errors (E3001–E3008)

These errors represent problems at the boundary between Nova and the
operating system — file access failures, network errors, database
errors, and format parse errors.

| Code | Name | Description |
|------|------|-------------|
| E3001 | file not found | File path does not exist or is inaccessible |
| E3002 | file read error | Failed to read file contents |
| E3003 | preprocessor error | Error during `#import`, `#define`, or `#include` processing |
| E3004 | parser init failed | Failed to initialize the parser state |
| E3005 | network error | HTTP request or network operation failed |
| E3006 | database error | SQL operation or database connection failed |
| E3007 | permission denied | Filesystem operation denied by OS permissions |
| E3008 | format/parse error | Data format codec failed to decode or encode |

### 2.4 Warnings (W1001–W1003)

Warnings are non-fatal diagnostics that highlight potential issues
without stopping execution. The warning code range is offset to
10001+ internally.

| Code | Name | Description |
|------|------|-------------|
| W1001 | unused variable | A declared dec variable is never read |
| W1002 | unreachable code | Code after a `return` or `break` can never execute |
| W1003 | shadowed variable | A new dec shadows an existing variable in an outer scope |

## 3. The Diagnostic Struct

Every error, warning, note, and help hint is represented by a
`NovaDiagnostic` struct. This is the universal currency of the error
system — created by the compiler or the runtime, enriched with
suggestion hints, and consumed by the renderer.

```c
typedef struct NovaDiagnostic {
    NovaDiagSeverity  severity;         /* ERROR, WARNING, NOTE, or HELP    */
    NovaErrorCode     code;             /* E1001, E2013, etc. (E0000 = none)*/
    const char       *filename;         /* Source file path                 */
    int               line;             /* 1-based line number              */
    int               column;           /* 1-based column (0 = unknown)     */
    int               end_column;       /* End column for underline range   */
    char              message[512];     /* Formatted diagnostic message     */
    const char       *source_line;      /* Pointer into source buffer       */
    int               source_line_len;  /* Length of source_line            */
    struct NovaDiagnostic *sub;         /* Chained sub-diagnostic (linked)  */
} NovaDiagnostic;
```

The `sub` field forms a singly-linked list of sub-diagnostics. An error
might carry one help hint, or two — a "did you mean?" suggestion
followed by a contextual note. The renderer walks the chain and prints
each sub-diagnostic with a `╰──` prefix.

The four severity levels have distinct rendering treatment:

| Severity | Label | Color | Purpose |
|----------|-------|-------|---------|
| `NOVA_DIAG_ERROR` | `error` | Bright red bold | Fatal errors |
| `NOVA_DIAG_WARNING` | `warning` | Amber/gold bold | Non-fatal warnings |
| `NOVA_DIAG_NOTE` | `note` | Sky blue bold | Informational context |
| `NOVA_DIAG_HELP` | `help` | Nova green bold | Actionable suggestions |

## 4. Error Catalog

Every error code maps to a catalog entry containing a short name and
an optional long-form explanation. The catalog is a static array of
`NovaErrorEntry` structs:

```c
typedef struct {
    NovaErrorCode  code;
    const char    *name;      /* 1-line description   */
    const char    *explain;   /* Multi-paragraph help  */
} NovaErrorEntry;
```

The catalog currently contains 57 entries covering all E1xxx, E2xxx,
E3xxx, and W1xxx codes. Short names appear in rendered diagnostics
(e.g., `error[E2013]: arithmetic type error`). Long explanations are
available via the `--explain` CLI flag.

### 4.1 The `--explain` Flag

Any error code can be passed to `nova --explain` for a detailed
tutorial:

```
$ nova --explain E2013

  E2013: arithmetic type error
  ──────────────────────────────

  An arithmetic operator (+, -, *, /, %, ^, //) was applied to
  values whose types don't support arithmetic. Both operands must
  be numbers (or have an appropriate __add, __sub, etc. metamethod).

  Common causes:
    - A string that looks numeric: "5" + 3
      Fix: use tonumber("5") + 3
    - A nil variable: x + 1 where x is uninitialized
      Fix: check that x is assigned before use
    - A boolean in arithmetic: true + 1
      Fix: booleans cannot be used as numbers in Nova

  Related: E2025 (bitwise on non-integer), E2015 (concatenation)
```

Explanations are compiled into the binary as static string literals.
They require no runtime file access, network access, or external
documentation.

## 5. Rendering Pipeline

The diagnostic renderer transforms a `NovaDiagnostic` into a colorful,
structured terminal output. The rendering process has six stages.

### 5.1 Header Line

The first line shows the severity label, error code, and short name:

```
error[E1014]: expected ')' after parameters
```

Severity labels are color-coded: red for errors, amber for warnings,
blue for notes, green for help. The error code in brackets uses Nova
green. The message text is post-processed by a colorizer that
highlights quoted tokens: `'expected'` tokens in green, `'got'` tokens
in red.

### 5.2 File Location

The second line shows the source file path and position:

```
  FILE: script.n  Ln 10, Col 5
```

File paths are rendered in bright white bold. Line and column numbers
use a silver accent color.

### 5.3 Context Window

The renderer extracts a window of source lines around the error —
3 lines before and 1 line after by default. Context lines are rendered
in a dimmed color; the error line is highlighted.

```
      │
   8  │  -- previous comment
   9  │  dec function calculate(a, b
  10  │      return a + b
      │  ────^^^^^^
  11  │  end
```

Line numbers are right-aligned in a sky blue gutter. The pipe
character `│` uses Nova green. The error line uses a light gray for
readability; surrounding context lines use a dimmed gray.

### 5.4 Caret Underline

Below the error line, a caret line marks the offending token. The
renderer estimates the token's width using a pattern-matching heuristic
that recognizes identifiers, numbers, strings, and multi-character
operators. The underline uses `^` characters preceded by `─` dashes
for the leading offset, all rendered in bright red.

```
      │  ────^^^^^^
```

### 5.5 Sub-Diagnostics

If the diagnostic has chained sub-diagnostics (help notes, suggestions),
each is rendered below with a `╰──` prefix and its own severity label
and color:

```
     ╰── help: did you mean 'print'?
     ╰── note: == and != work across types (different types are never equal)
```

Help hints use Nova green; notes use sky blue. Sub-diagnostics are
rendered recursively — a sub-diagnostic can itself carry nested
sub-diagnostics, though in practice the depth rarely exceeds one.

### 5.6 Stack Traceback

For runtime errors, a stack traceback is printed after the diagnostic.
The VM unwinds the call stack and formats each frame with its source
location:

```
stack traceback:
        script.n:42: in function 'calculate'
        script.n:10: in main chunk
```

### 5.7 Full Example

Putting it all together, here is what a developer sees when they
misspell a function name:

```
[E2008]: attempt to call a nil value
  FILE: =(eval)  Ln 1
     │
 1   │ prnt("hello")
     ╰── help: did you mean 'print'?

stack traceback:
        =(eval):1: in main chunk
```

And here is an arithmetic type error with a help note:

```
[E2013]: attempt to perform '+' on string and integer
  FILE: =(eval)  Ln 1
     │
 1   │ dec x = "hello" + 1
     ╰── help: use tonumber() to convert a string to a number, or '..' to concatenate strings

stack traceback:
        =(eval):1: in main chunk
```

## 6. Error Classification

Errors are classified — mapped from raw failure conditions to specific
error codes — at three distinct points in the pipeline. Each
classification layer adds precision.

### 6.1 Compile-Time Classification

The compiler classifies errors at the point they are detected. An
internal function `novai_infer_compile_code()` pattern-matches the
error message text against known categories:

| Pattern | Code |
|---------|------|
| `"too many registers"` | E1004 |
| `"too many nested scopes"` | E1005 |
| `"too many dec"` | E1003 |
| `"too many nested functions"` | E1006 |
| `"invalid assignment"` | E1007 |
| `"break"` or `"continue"` | E1008 |
| `"duplicate label"` | E1009 |
| `"unresolved goto"` | E1010 |
| `"too many arguments"` | E1011 |
| `"constant overflow"` | E1013 |
| `"out of memory"` | E1017 |
| `"module name"` | E1020 |

If no pattern matches, the error is reported with code E0000 (message
only, no bracketed code in the output).

### 6.2 Runtime Auto-Classification

For runtime errors raised through the standard error path, an
auto-classifier `novai_auto_classify()` in the VM examines the error
message text and assigns a code:

| Pattern | Code |
|---------|------|
| `"bad argument"` + `"value expected"` | E2022 |
| `"bad argument"` | E2021 |
| `"out of memory"` or `"memory allocation"` | E2003 |
| `"module"` + `"not found"` | E2023 |
| `"sql."` prefix | E3006 |
| `"net."` or `"net:"` prefix | E3005 |
| `"cannot open"` / `"cannot read"` / `"cannot seek"` | E3007 |
| `".decode:"` / `".encode:"` / `".load:"` | E3008 |

This classifier covers all 176 standard library call sites. Any
function in the standard library that calls `novai_error()` with a
message matching one of these patterns will automatically receive the
correct fine-grained code — without the library function needing to
know about the error code system at all.

### 6.3 Explicit Classification via `diag_code`

For the most critical error paths — metamethod dispatch, operator
type errors — the code is set explicitly. The VM struct carries a
`diag_code` field that error-raising functions can populate:

```c
vm->diag_code = NOVA_E2013;  /* arithmetic type error */
novai_error(vm, NOVA_VM_ERR_RUNTIME, "...");
```

The function `nova_vm_raise_error_ex()` wraps this pattern, accepting
both a VM error type and a diagnostic code:

```c
nova_vm_raise_error_ex(vm, NOVA_VM_ERR_TYPE, NOVA_E2013,
    "attempt to perform '+' on %s and %s", ltype, rtype);
```

When the CLI reports the error, it prefers `vm->diag_code` if set,
falling through to auto-classification only as a fallback.

## 7. Runtime Error Enrichment

Nova doesn't just classify errors — it enriches them with precise
type information and operator symbols extracted from the execution
context. This enrichment happens at the point of failure, before the
error propagates.

### 7.1 Type Names in Error Messages

Every runtime type error message includes the actual type names of the
operands. When you write `"hello" + 1`, the error message says:

```
attempt to perform '+' on string and integer
```

Not the generic `"attempt to perform arithmetic on a string value"`.
The VM calls `nova_vm_typename()` to extract human-readable type names
(`"nil"`, `"boolean"`, `"integer"`, `"number"`, `"string"`, `"table"`,
`"function"`, `"userdata"`, `"coroutine"`) and formats them into the
error message at the point of failure.

### 7.2 Operator Symbols in Error Messages

For arithmetic, comparison, and bitwise errors, the error message
includes the actual operator symbol. An internal function
`novai_opcode_symbol()` maps the failing opcode to its source-level
representation:

| Opcode Family | Symbol |
|---------------|--------|
| ADD, ADDI, ADDK | `+` |
| SUB, SUBK | `-` |
| MUL, MULK | `*` |
| DIV, DIVK | `/` |
| MOD, MODK | `%` |
| POW | `^` |
| IDIV | `//` |
| BAND | `&` |
| BOR | `\|` |
| BXOR | `~` |
| SHL | `<<` |
| SHR | `>>` |
| UNM | `-` (unary) |
| BNOT | `~` (bitwise NOT) |
| STRLEN | `#` |

This means the error message for `"hello" ^ 2` says `attempt to
perform '^' on string and integer` — the developer sees the exact
operator that failed, not a generic "arithmetic error".

### 7.3 Enrichment by Error Class

Each category of runtime error is enriched with specific contextual
information:

**Arithmetic (E2013):** Operator symbol, both operand types.
`"attempt to perform '+' on string and integer"`

**Comparison (E2014):** Both operand types, comparison operator.
`"attempt to compare string with integer using '<'"`

**Concatenation (E2015):** Both operand types.
`"attempt to concatenate string and boolean"`

**Length (E2016):** Operand type, expected types.
`"attempt to get length of a nil value (string or table expected)"`

**Negation (E2017):** Operand type, expected type.
`"attempt to negate a string value (number expected)"`

**Index (E2018):** Indexed value's type.
`"attempt to index a nil value"`

**Call (E2008):** Called value's type.
`"attempt to call a integer value"`

**Bitwise (E2025):** Operator symbol, operand type.
`"attempt to perform '&' on string and number"`

## 8. The Suggestion Engine

The suggestion engine is the most distinctive component of Nova's error
system. It is implemented in `nova_suggest.c` and draws on techniques
from Natural Language Processing — the same algorithmic family that
powers Nova's built-in NLP library.

### 8.1 Architectural Overview

The engine has three layers:

1. **Fuzzy name matching** — Levenshtein edit distance finds the
   closest known name when a function or method call fails on nil.

2. **KWIC source analysis** — Key Word In Context extractors pull
   identifiers from the source line to understand what the developer was
   trying to do.

3. **Per-error-class hint generators** — Eleven specialized functions
   produce actionable sub-diagnostics for each category of error.

The engine is a pure diagnostic utility. It depends only on
`nova_error.h` and standard C — it does not link against the VM, the
compiler, or any external library.

### 8.2 Levenshtein Edit Distance

The `novai_levenshtein()` function computes the minimum number of
single-character insertions, deletions, and substitutions needed to
transform one string into another. It uses the classic dynamic
programming algorithm with a single-row space optimization:

- **Space complexity:** O(min(m, n)) — one row, swapped so the shorter
  string determines the allocation size.
- **Time complexity:** O(m × n) for strings of length m and n.
- **Case-insensitive:** All comparisons use `tolower()`.

The public function `nova_suggest_name()` wraps the distance
computation with candidate iteration, exact-match skipping (an exact
match is not a typo), and length-difference pre-filtering.

#### Adaptive Thresholds

Short identifiers are held to a tighter matching standard. A 3-letter
name like `len` should not suggest `log` (distance 2), because at that
length almost every short word is within distance 2 of another. The
engine uses an adaptive threshold:

| Input Length | Max Edit Distance |
|-------------|-------------------|
| 1–3 chars | 1 |
| 4–5 chars | 2 |
| 6+ chars | 3 (full limit) |

This prevents false positives on short names while still catching
substantial typos on longer identifiers.

### 8.3 Name Dictionaries

The engine maintains three static dictionaries of known names:

**Global names** — Top-level functions and module tables that are always
available in Nova:

```
echo  print  printf  sprintf  fprintf  type  tostring  tonumber
error  assert  pcall  xpcall  rawget  rawset  rawlen  rawequal
select  pairs  ipairs  next  unpack  require  dofile  load
setmetatable  getmetatable
math  string  table  io  os  fs  coroutine  debug  tools  async
package  net  sql  nlp  data
```

**Method names** — Functions available on module tables and through
method syntax (`:`, `.`):

```
len  sub  upper  lower  rep  find  format  gsub  match  gmatch
byte  char  reverse  insert  remove  sort  concat  move  pack
unpack  abs  ceil  floor  sqrt  sin  cos  tan  log  exp  random
max  min  open  close  read  write  lines  execute  capture
getenv  setenv  clock  time  date  sleep  platform  exists
isfile  isdir  list  walk  glob  mkdir  copy  stat  create
resume  yield  wrap  status  traceback  getinfo  getlocal
sethook  tokenize  stem  distance  similarity  fuzzy  freq
tfidf  ngrams  kwic  sentences  summarize  normalize  ...
```

**Module names** — Known importable module names for `require()` and
`#import`:

```
math  string  table  io  os  fs  coroutine  debug  tools  async
net  sql  nlp  data  json  csv  nini  package
```

### 8.4 KWIC Source Analysis

KWIC — Key Word In Context — is a concordance technique from corpus
linguistics. In its original form, it extracts a window of text
surrounding each occurrence of a keyword to reveal usage patterns. Nova
adapts this idea to source code: when an error occurs, the engine
extracts identifiers from the source line to understand what the
developer was trying to do.

Three extractors provide source context:

**Call name extraction** (`novai_extract_call_names()`) scans the
source line for identifiers that appear immediately before `(`. These
are call targets. For `prnt("hello")`, it extracts `prnt`. For
`math.sqr(x)`, it extracts both `math` and `sqr`.

**Method name extraction** (`novai_extract_method_names()`) scans for
identifiers after `.` or `:`. These are field-access or method-call
targets. For `s:uper()`, it extracts `uper`. For `math.sqr(x)`, it
extracts `sqr`.

**Pattern detection** (`novai_line_contains()`) checks whether the
source line contains specific substrings that imply known mistake
patterns. For example, the substring `require` suggests the error may
be related to module loading. The substring `[#` suggests a potential
0-based indexing mistake (since `t[#t]` is out of bounds in Nova's
0-based arrays).

### 8.5 Hint Generators

The engine includes eleven specialized hint generators, one per
runtime error class. Each generator analyses the error message and
source context to produce zero or more `NovaDiagnostic` sub-hints.

| Error Code | Hint Generator | Suggestions Produced |
|------------|---------------|---------------------|
| E2008 | `novai_hint_call` | "did you mean 'X'?" for misspelled functions/methods; nil variable notes; `require()` context detection; `__call` metamethod suggestion for non-function calls |
| E2013, E2025 | `novai_hint_arith` | `tonumber()` for strings; nil initialization checks; boolean/table-specific guidance; metamethod suggestions for tables |
| E2014 | `novai_hint_compare` | Same-type ordering rule; `==`/`!=` cross-type note |
| E2015 | `novai_hint_concat` | `tostring()` conversion; nil operand detection |
| E2016 | `novai_hint_length` | `#` requires string or table; `type()` check suggestion |
| E2017 | `novai_hint_negate` | `tonumber()` for strings; nil checks; length operator confusion |
| E2018 | `novai_hint_index` | Nil initialization guidance; `require()` context; 0-based index pitfall detection (`[#t]` pattern) |
| E2020 | `novai_hint_for` | All three `for` bounds must be numbers; `tonumber()` for string bounds |
| E2021 | `novai_hint_arg` | Type-specific conversion hints (`tostring()`, `tonumber()`, `{}`); method name typo detection |
| E2022 | `novai_hint_argcount` | Missing required argument reminder |
| E2023 | `novai_hint_module` | Module name fuzzy matching ("did you mean 'math'?"); `package.path` check |

### 8.6 The "Did You Mean?" Flow

The highest-impact suggestion is the "did you mean?" hint for
misspelled names. Here is the complete flow for `prnt("hello")`:

1. The VM encounters `OP_CALL` on a nil value. It raises E2008:
   `"attempt to call a nil value"`.

2. The CLI creates a `NovaDiagnostic` and calls
   `nova_suggest_runtime_hints()`.

3. The dispatcher routes to `novai_hint_call()` based on code E2008.

4. The hint generator detects "nil" in the message. Source context is
   available, so it proceeds to KWIC analysis.

5. `novai_extract_method_names()` finds no matches (no `.` or `:` in
   the source line). `novai_extract_call_names()` identifies `prnt`
   before the `(`.

6. `nova_suggest_name("prnt", 4, novai_global_names, -1, 3)` is
   called. The adaptive threshold for a 4-char name allows edit
   distance up to 2.

7. `novai_levenshtein("prnt", "print")` returns 1 (one insertion).
   This is the best match in the dictionary.

8. The hint generator writes a sub-diagnostic:
   `severity=HELP, message="did you mean 'print'?"`.

9. The CLI attaches the hint to the parent diagnostic and emits it.

The developer sees:

```
[E2008]: attempt to call a nil value
  FILE: =(eval)  Ln 1
     │
 1   │ prnt("hello")
     ╰── help: did you mean 'print'?
```

### 8.7 Method-First Resolution

When a call error occurs on a name accessed through `.` or `:`, the
engine prioritizes method names over global names. This prevents
false matches: `math.sqr(9)` should suggest `sqrt` (a method name),
not `sql` (a global name that happens to be closer in edit distance
to `sqr`).

The resolution order is:

1. Extract method names after `.` or `:` — match against the method
   dictionary.
2. If no method match, extract bare call names before `(` — match
   against the global dictionary.
3. If no global match, try the bare call names against the method
   dictionary as a fallback.

This ensures that `math.sqr(9)` → `"did you mean 'sqrt'?"` and
`prnt("hello")` → `"did you mean 'print'?"` both work correctly.

## 9. Integration: The Create–Attach–Emit Pattern

The error system is wired into the Nova CLI through a three-phase
pattern that separates diagnostic creation, hint enrichment, and
rendering.

```c
/* Phase 1: CREATE the primary diagnostic */
NovaDiagnostic diag;
nova_diag_create(&diag, NOVA_DIAG_ERROR, code,
                 filename, line, column, "%s", message);

/* Phase 2: GENERATE and ATTACH hints */
NovaDiagnostic hints[4];
int nhints = nova_suggest_runtime_hints(
    hints, 4, (int)code, message,
    diag.source_line, diag.source_line_len);
for (int h = 0; h < nhints; h++) {
    nova_diag_attach(&diag, &hints[h]);
}

/* Phase 3: EMIT with full rendering */
nova_diag_emit(&diag);
```

`nova_diag_create()` fills the diagnostic struct and extracts the
source line from the global source context (set by
`nova_diag_set_source()` before compilation/execution). The extracted
source line is then available to the suggestion engine in Phase 2.

`nova_suggest_runtime_hints()` writes up to 4 sub-diagnostics into a
caller-owned array. Each hint is a complete `NovaDiagnostic` with
severity `HELP` or `NOTE`, code `E0000`, and a formatted message.

`nova_diag_attach()` chains each hint into the parent's `sub` linked
list. `nova_diag_emit()` renders the complete tree to stderr.

A convenience function `nova_diag_report()` exists for simple cases
where no hints are needed — it performs create-and-emit in one call.

### 9.1 Source Context Lifecycle

Source context must be set before compilation or execution and cleared
afterward:

```c
NovaDiagSource src = { filename, source_text, source_len };
nova_diag_set_source(&src);   /* set before execution */

/* ... compile and execute ... */

nova_diag_set_source(NULL);   /* clear after completion */
```

The global source pointer is not copied — the source buffer must remain
valid for the duration. When executing from bytecode files (`.no`),
source context is not available and hints gracefully degrade to
message-only analysis.

## 10. Color System

Nova uses a carefully designed ANSI 256-color palette organized into
four categories. All color output is conditioned on a global flag
that respects terminal capabilities and the `NO_COLOR` standard.

### 10.1 Brand Colors

| Name | ANSI Code | Usage |
|------|-----------|-------|
| Hunter Green | `38;5;22` | Brand identity, subtle accents |
| Nova Green | `1;38;5;40` | Success states, help text, pipe characters |
| Emerald | `38;5;35` | Secondary accents |
| Forest Green | `38;5;28` | Dimmed green tones |
| Lime | `38;5;118` | Bright highlights |

### 10.2 Severity Colors

| Severity | Color | ANSI Code |
|----------|-------|-----------|
| Error | Bright red bold | `1;38;5;196` |
| Warning | Amber/gold bold | `1;38;5;214` |
| Note | Sky blue bold | `1;38;5;75` |
| Help | Nova green bold | `1;38;5;40` |

### 10.3 Structural Colors

| Element | Color | Usage |
|---------|-------|-------|
| Path | Bright white bold | File paths in headers |
| Line number | Sky blue | Gutter line numbers |
| Gutter | Medium gray | Pipe characters in gutter |
| Caret | Bright red | Underline markers (^) |
| Source | Light gray | Error line text |
| Error code | Nova green | `[E1001]` bracket codes |
| Context line | Dimmed gray | Lines before/after error |
| Expected token | Nova green | Quoted tokens in "expected 'X'" messages |
| Got token | Bright red | Quoted tokens in "got 'Y'" messages |

### 10.4 Color Control

Colors are enabled by default when stderr is a terminal (detected via
`isatty()`). Two override mechanisms exist:

- **Environment variable:** Setting `NO_COLOR` to any non-empty value
  disables all color output, following the [no-color.org](https://no-color.org)
  standard.
- **API function:** `nova_diag_set_color(0)` disables color
  programmatically; `nova_diag_set_color(1)` enables it.
